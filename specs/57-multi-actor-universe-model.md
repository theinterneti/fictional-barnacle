## 1. Purpose

S57 promotes the v1 stub S21 (Co-Authoring / Shared Worlds) into a concrete
specification. It defines the **world-model changes** required to support
multiple actors (player-controlled) simultaneously occupying a single universe.

S57 is the data model and engine layer. S58 defines how simultaneous turns
are resolved. S59 defines the transport (WebSocket).

In v1, each universe had exactly one active player-actor at a time. S57 relaxes
this to: a universe may have **1..N** player-actors, where N is bounded by
`MAX_ACTORS_PER_UNIVERSE` (default: 4 in v4; configurable per universe manifest).

---

## 2. Actor Roster

A universe instance now maintains an **actor roster**: an ordered list of actors
currently active in the universe.

```python
class UniverseActorRoster(BaseModel):
    universe_id: str
    actors: list[ActorSlot]
    max_actors: int  # from manifest, default 4

class ActorSlot(BaseModel):
    actor_id: uuid.UUID
    player_id: uuid.UUID
    joined_at: datetime
    status: Literal["active", "idle", "departing"]
    current_location_id: str | None
    last_turn_at: datetime | None
```

The roster is stored in Redis (keyed `tta:universe:{id}:roster`) with a
sliding TTL of `UNIVERSE_ROSTER_TTL_SECONDS` (default: 900 s) refreshed on
every actor-state write (per S12 FR-12.13), and materialized as part of
the universe's world-state in PostgreSQL.

---

## 3. Per-Actor Narrative Streams

Each actor maintains their own narrative stream: a sequence of turns, each
with a dedicated context window. The world-state is shared; the narrative
perspective is per-actor.

This means:
- Each actor's LLM call receives context filtered to their perspective
  (their memories, their location, their recent turns)
- The **world-state** (NPC positions, item locations, environmental events)
  is shared and mutated by any actor's actions
- A world-state change by Actor A is visible to Actor B's next turn

---

## 4. Shared World-State

The shared world-state is the existing S04 world model (Neo4j graph). No
structural changes are needed. The change is that *multiple actors may read
and write to the same world-state concurrently*.

Concurrency control:
- **Optimistic locking**: world-state writes include a `version` token.
  If the version has changed since the actor's last read, the write is
  retried after re-reading the current state.
- **S58** handles the case where two actors' turns produce conflicting
  world-state mutations.

## 5. Functional Requirements

### FR-57.01 — Actor Roster Management

An actor joins a universe's roster via `ActorRosterService.join(actor_id, universe_id)`.
A join fails if:
- The roster is at `max_actors` capacity → `RosterFullError`
- The actor is already in a different universe → `ActorAlreadyElsewhereError`
  (actors must travel via S51, not join directly)

An actor leaves via `ActorRosterService.depart(actor_id, universe_id)`. Departure
persists the actor's local state to PostgreSQL.

### FR-57.02 — Turn Pipeline Per Actor

Each active actor may submit a turn independently. The turn pipeline is invoked
once per actor-turn. Two actors submitting turns simultaneously triggers S58
conflict resolution.

### FR-57.03 — World-State Visibility

An actor's turn context includes a "shared world digest": a short summary of
recent world-state changes caused by other actors, generated by a dedicated
LLM call (the "world update summarizer"). This keeps actors informed without
exposing raw internal state.

### FR-57.04 — Actor Presence in NPC Interactions

NPCs are aware of all actors currently in their location. The NPC generation
prompt includes a `present_actors` list so the LLM can have NPCs acknowledge
and respond to multiple characters being present simultaneously.

### FR-57.05 — Idle Timeout

An actor slot transitions to `idle` after `actor_idle_timeout` (default: 10
minutes) of no turn activity. After `actor_removal_timeout` (default: 30 minutes)
idle, the actor is automatically departed and their slot freed.

## 6. Acceptance Criteria (Gherkin)

```gherkin
Feature: Multi-Actor Universe Model

  Scenario: AC-57.01 — Two actors can join the same universe
    Given Universe X is ACTIVE with max_actors = 4
    When Actor A and Actor B both join Universe X
    Then the roster contains both actors
    And both can take independent turns

  Scenario: AC-57.02 — Roster at capacity rejects new actor
    Given Universe X has 4 actors and max_actors = 4
    When Actor E attempts to join
    Then a RosterFullError is returned

  Scenario: AC-57.03 — World-state change by Actor A is visible to Actor B
    Given Actor A picks up an item in location L
    When Actor B's next turn processes
    Then Actor B's world digest mentions the item is no longer in location L

  Scenario: AC-57.04 — NPC acknowledges multiple actors
    Given Actor A and Actor B are in the same location as NPC Mira
    When Actor A speaks to Mira
    Then Mira's response may address the presence of Actor B
```

---

## 7. Out of Scope

- Real-time voice or text chat between actors (S59 transport only).
- Player-visible actor presence lists (UX design, not engine).
- Actor teaming mechanics (narrative only; no game-mechanic grouping).
