## 1. Purpose

S51 defines the mechanic by which an actor moves from one loaded universe
(Universe A) to another (Universe B). It answers three questions:
1. **When** can travel occur? (trigger conditions)
2. **What** survives the crossing? (character-state transfer rules)
3. **How** does the receiving universe orient the arriving actor? (onboarding)

S31 established that actor IDs are universe-agnostic and character state is
universe-scoped. S51 operationalizes this by defining which state is
"core" (travels with the actor) versus "local" (left behind in Universe A).

---

## 2. Character State Transfer Model

State is divided into three tiers:

| Tier | Contents | On Crossing |
|---|---|---|
| **Core** | `actor_id`, `display_name`, `pronouns`, identity-level attributes | Always preserved |
| **Echo** | A curated subset of MemoryRecords flagged as `cross_universe: true` | Carried as latent memories; may be referenced but not assumed true in B |
| **Local** | Current location, social relationships, inventory, local world-state | Stays in Universe A (persisted); reset in Universe B |

Echo memories are surfaced to the LLM with a narrative framing:
*"You remember fragments of another place: [echo summary]"*. They inform
character voice without imposing Universe A's facts on Universe B.

---

## 3. Trigger Conditions

Travel can be initiated by three paths:

| Trigger | Initiator | Description |
|---|---|---|
| `narrative_portal` | LLM | World-model event that opens a crossing; generated by the LLM during a session turn when narrative conditions are met |
| `player_ritual` | Player | Player-initiated via a specific in-game action sequence defined by the universe manifest |
| `admin_transfer` | Admin API | POST /admin/actors/{id}/transfer — used for testing and support |

## 4. Functional Requirements

### FR-51.01 — Travel Transaction

A `CrossUniverseTravelTransaction` wraps the full crossing sequence:
1. Validate that Universe B is ACTIVE (or initiate load via S50)
2. Depart: persist actor's local state in Universe A; mark actor as `in_transit`
3. Transfer: project core state + echo memories into Universe B's identity format
4. Arrive: create actor's local state in Universe B; mark actor as `in_universe:B`
5. Emit a `travel_completed` event on the S54 substrate (both universes subscribed)

If any step fails, the transaction rolls back to the last stable state. An actor
MUST NOT be simultaneously `in_transit` and processing turns.

### FR-51.02 — State Projection

State projection from Universe A → Universe B involves:
- Core attributes: copied verbatim
- Echo memories: copied as MemoryRecords with `source_universe_id` annotation
  and `importance_score` halved (echo memories fade with time in Universe B)
- Inventory: each item is passed through a `ThematicTranslator` that maps
  the item's S39 archetype to the nearest equivalent in Universe B's vocabulary.
  If no equivalent exists, the item becomes an unrecognized artefact with
  a description derived from its original archetype.

### FR-51.03 — Arrival Onboarding

On arrival, Universe B's turn pipeline produces a **crossing narration turn**:
a special first-turn narrative that:
- Acknowledges the actor's arrival in the new universe
- Grounds the actor in their new location (determined by Universe B's arrival zone)
- Weaves in one or two echo memories as narrative texture
- Does NOT assume facts from Universe A are true in Universe B

The crossing narration is generated by the LLM with a dedicated system prompt
that includes the arrival context and echo memory fragments.

### FR-51.04 — Concurrent Universe Requirement

Travel requires both Universe A and Universe B to be simultaneously ACTIVE
(via S50). The travel transaction reserves space in Universe B before
departing from Universe A. If Universe B cannot load within 30 seconds,
the travel attempt fails with `UniverseUnavailableError`.

### FR-51.05 — Return Travel

An actor may return to Universe A from Universe B using the same trigger
mechanisms. When returning, the actor resumes from their persisted local
state in Universe A (not a fresh arrival). Echo memories accumulated in
Universe B are added to the returning actor's echo pool in Universe A.

## 5. Acceptance Criteria (Gherkin)

```gherkin
Feature: Cross-Universe Travel

  Scenario: AC-51.01 — Core identity survives crossing
    Given actor A with display_name = "Mira" in Universe X
    When the actor travels to Universe Y
    Then actor A exists in Universe Y with display_name = "Mira"
    And actor_id is unchanged

  Scenario: AC-51.02 — Local state is left behind
    Given actor A is at location L in Universe X with social relationship R
    When the actor travels to Universe Y
    Then L and R are preserved in Universe X
    And the actor has no location or relationships in Universe Y at arrival

  Scenario: AC-51.03 — Echo memories travel with halved importance
    Given actor A has a MemoryRecord (cross_universe=true, importance_score=0.8)
    When the actor travels to Universe Y
    Then Universe Y contains the memory with importance_score = 0.4
    And the memory has source_universe_id = Universe X

  Scenario: AC-51.04 — Crossing narration is generated on arrival
    Given actor A arrives in Universe Y
    When the arrival turn is processed
    Then a crossing narration is emitted that references arrival location
    And references at least one echo memory fragment
    And does not assert facts from Universe X as true

  Scenario: AC-51.05 — Travel fails if Universe B cannot load
    Given Universe B cannot reach ACTIVE state within 30 seconds
    When actor A attempts to travel to Universe B
    Then a UniverseUnavailableError is returned
    And actor A remains active in Universe A
```

---

## 6. Out of Scope

- Simultaneous occupation of two universes (an actor is in exactly one universe
  at a time; this constraint is enforced by S31).
- Group travel (all actors travel independently; coordinated travel is a
  narrative possibility, not a system primitive).
- History of all past universes visited (echo memory pool handles this
  implicitly; no explicit travel log is required).
