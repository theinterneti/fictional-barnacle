## How to Read This Spec

This is a **functional specification** — it describes *what* the system does, not *how*
it's built. It is one half of a testable contract:

- **This spec** (the "what") defines behavior, acceptance criteria, and boundaries
- **The technical plan** (the "how") will define architecture, stack, and implementation
- **Tasks** will decompose both into small, reviewable chunks of work

Acceptance criteria use **Gherkin syntax** (Given/When/Then) so they can be directly
executed as automated BDD tests using frameworks like Behave or pytest-bdd.

## 1. Purpose

This spec defines the **v1 minimum** content moderation layer for TTA. It establishes
input screening, output validation, and pipeline hook points that will be extended by
S19 (Crisis & Content Safety) in future releases.

TTA generates narrative content via LLMs. Without moderation, the system could produce
or accept content that is harmful, inappropriate, or inconsistent with the game's
therapeutic intent. This spec defines the guardrails.

### Relationship to S19

S19 (Crisis & Content Safety) is a **future stub** that defines the full safety system
TTA will eventually need: crisis detection, real-time safety scoring, escalation to human
reviewers, and integration with safety reporting APIs. S19 explicitly lists six **v1
constraints** — architectural decisions v1 must make to accommodate the future feature.

This spec (S24) fulfills those six constraints:

| S19 v1 Constraint | S24 Fulfillment |
|---|---|
| Pipeline hook points for content analysis | §3.1 — Moderation hook in turn pipeline |
| Ability to interrupt a streaming response | §3.3 — Stream interruption on flagged content |
| Session flagging for human review | §3.4 — Flag system for operator review |
| Audit trail for moderation actions | §3.5 — Moderation event logging |
| Content classification hooks | §3.2 — Content classification categories |
| Graceful degradation when moderation service is unavailable | §3.6 — Moderation unavailability behavior |

### Design values applied

| Value | Implication |
|---|---|
| **Fun** | Moderation should be invisible when content is appropriate. It should not add noticeable latency or interrupt the narrative flow. |
| **Safety** | The game has a therapeutic context. Content that could cause psychological harm must be caught, even at the cost of some false positives. |
| **Transparency** | Players should understand that content moderation exists and why. Operators should see exactly what was moderated and why. |
| **Craftsmanship** | Moderation is a first-class pipeline stage, not an afterthought filter. It has its own tests, metrics, and error handling. |

## 2. User Stories

### US-24.1 — Player is protected from harmful generated content
> As a **player**, I want the game to never show me graphic violence, explicit sexual
> content, or self-harm instructions generated by the LLM, so that the game remains a
> safe therapeutic experience.

### US-24.2 — Player's input is screened for inappropriate content
> As a **player**, I want the game to gently redirect me if I submit content that the
> system cannot safely process (e.g., requests for harmful content), so that the
> interaction remains constructive.

### US-24.3 — Operator reviews flagged content
> As an **operator**, I want to see a log of all moderation actions (blocks, flags,
> redirections) with context, so that I can tune moderation sensitivity and identify
> patterns of misuse.

### US-24.4 — Developer adds moderation to the pipeline
> As a **developer**, I want content moderation to be a standard pipeline stage with clear
> input/output contracts, so that I can test it in isolation and extend it for future
> safety features.

### US-24.5 — System degrades gracefully when moderation fails
> As an **operator**, I want the system to continue serving content (with appropriate
> logging and alerting) if the moderation service is temporarily unavailable, rather than
> blocking all gameplay.

## 3. Functional Requirements

### 3.1 — Moderation Pipeline Integration

**FR-24.01**: Content moderation SHALL be a distinct stage in the turn processing
pipeline (per S08), positioned:
- **Input moderation**: After player input parsing, before LLM context assembly
- **Output moderation**: After LLM response generation, before SSE streaming to player

**FR-24.02**: Each moderation stage SHALL receive the content to evaluate and return one
of three verdicts:
- `pass` — Content is acceptable. Proceed normally.
- `flag` — Content is borderline. Proceed, but log the event and flag for operator review.
- `block` — Content is unacceptable. Do not proceed. Return a safe alternative.

**FR-24.03**: Moderation verdicts SHALL include:
- `verdict`: pass | flag | block
- `category`: The content classification that triggered the verdict (see §3.2)
- `confidence`: A float between 0.0 and 1.0 indicating how confident the classification is
- `reason`: A brief, machine-readable explanation (e.g., "self_harm_instruction")

### 3.2 — Content Classification

**FR-24.04**: The system SHALL classify content into the following categories:

| Category | Description | Default action | Override-able? |
|---|---|---|---|
| `safe` | Content is appropriate for the game's context | pass | No |
| `mild_violence` | Fantasy/narrative violence within genre norms | pass | Yes (flag) |
| `graphic_violence` | Detailed depictions of gore, torture, dismemberment | block | No |
| `sexual_content` | Explicit sexual descriptions | block | No |
| `self_harm` | Instructions or encouragement of self-harm | block | No |
| `hate_speech` | Slurs, discrimination, dehumanization | block | No |
| `dangerous_activity` | Instructions for real-world dangerous or illegal activity | block | No |
| `personal_info` | Content containing PII (names, addresses, numbers) | flag | Yes (block) |
| `off_topic` | Content unrelated to the game (e.g., political debate, real-world queries) | flag | Yes (block) |
| `prompt_injection` | Attempts to manipulate system prompts or override instructions | block | No |

**FR-24.05**: Default actions (pass/flag/block) SHALL be configurable per-category via
application configuration. Non-overridable categories (`graphic_violence`, `sexual_content`,
`self_harm`, `hate_speech`, `dangerous_activity`, `prompt_injection`) SHALL always block
regardless of configuration.

### 3.3 — Stream Interruption

**FR-24.06**: When output moderation detects blocked content DURING SSE streaming (the
LLM has already begun generating tokens), the system SHALL:
1. Stop forwarding tokens to the player immediately
2. Send a narrative-appropriate redirection message (e.g., "The story shifts away
   from that path…")
3. Send an SSE `moderation` event indicating content was redirected
4. Log the full generated content (including blocked tokens) for operator review
5. Mark the turn as `moderated` in persistence

**FR-24.07**: Stream interruption SHALL occur within 2 tokens of detecting blocked
content. The system SHOULD evaluate content moderation in a sliding window over the
streaming token buffer, not wait for the full response.

**FR-24.08**: The redirection message SHALL be a pre-authored fallback (from prompt
templates, per S09) that narratively pivots the scene without breaking immersion.

### 3.4 — Flagging System

**FR-24.09**: Every moderation action (pass, flag, block) SHALL be recorded with:
- `moderation_id`: Unique identifier
- `turn_id`: The turn being moderated
- `game_id`: The game context
- `player_id`: The player (anonymized in external logs per S17)
- `stage`: "input" or "output"
- `content_hash`: SHA-256 hash of the moderated content (not the content itself in logs)
- `verdict`: pass | flag | block
- `category`: Content classification
- `confidence`: Classification confidence
- `timestamp`: ISO 8601

**FR-24.10**: Moderation records with verdict `flag` or `block` SHALL be queryable by
operators via the admin API (per S26). Operators SHALL be able to:
- List flagged content by time range, category, or player
- View the original content (access-controlled, audited)
- Mark a flag as "reviewed" or "escalated"

**FR-24.11**: If a player accumulates N blocked actions within M minutes (configurable,
default N=5, M=10), the system SHALL automatically flag the session for operator review
and log a high-severity moderation alert.

### 3.5 — Audit Trail

**FR-24.12**: Moderation events SHALL be logged via the standard structured logging
system (per S15/S23) with event type `moderation_action`.

**FR-24.13**: The moderation audit trail SHALL be separate from the general application
log. It SHALL be queryable independently and retained per the data retention schedule
in S17.

**FR-24.14**: The audit trail SHALL record the full moderated content (input or output
text) in a separate, access-controlled store. General logs SHALL reference the
`moderation_id` and `content_hash` only — never the content itself.

### 3.6 — Graceful Degradation

**FR-24.15**: If the moderation system is unavailable (e.g., external moderation API
down or moderation model fails to load), the system SHALL:
1. Continue processing turns WITHOUT moderation (fail-open for v1)
2. Log every unmoderated turn at WARN level
3. Expose the moderation service status in `/api/v1/health` (per S23 FR-23.23)
4. Alert the operator immediately

**FR-24.16**: The decision to fail-open vs. fail-closed SHALL be configurable
(`TTA_MODERATION_FAIL_MODE=open|closed`). Default is `open` for v1 to prioritize
gameplay availability. Future releases (per S19) may default to `closed`.

> **Design note**: Fail-open in v1 is a deliberate tradeoff. TTA targets therapeutic
> contexts where blocking all gameplay due to a moderation outage could itself be
> harmful (e.g., if a player is relying on the game for emotional regulation). S19
> will introduce more sophisticated fallback strategies.

### 3.7 — Player Communication

**FR-24.17**: When player input is blocked, the system SHALL return a response that:
- Does NOT reveal the specific moderation category or classification confidence
- Gently redirects the player (e.g., "Let's take the story in a different direction.
  What else would you like to do?")
- Does NOT use accusatory language ("Your message was blocked")

**FR-24.18**: When output is blocked during streaming, the redirection message SHALL be
in the narrative voice of the current scene, maintaining immersion (per FR-24.08).

**FR-24.19**: The system SHALL NOT inform the player that their content was "moderated."
From the player's perspective, the narrative simply takes a different direction.

## 4. Non-Functional Requirements

### NFR-24.1 — Moderation Latency
**Category**: Performance
**Target**: Input moderation SHALL complete within 100ms (p95). Output moderation on
streaming content SHALL evaluate tokens within 50ms (p95) to avoid visible streaming
delays.

### NFR-24.2 — False Positive Rate
**Category**: Quality
**Target**: Moderation SHALL have a false positive rate below 5% for the `block` verdict
on player input. Fantasy violence within genre norms ("I attack the dragon with my
sword") SHALL NOT be blocked.

### NFR-24.3 — Classification Coverage
**Category**: Quality
**Target**: The moderation system SHALL correctly classify content for all categories in
FR-24.04 with recall ≥ 90% for safety-critical categories (`self_harm`, `hate_speech`,
`dangerous_activity`).

### NFR-24.4 — Audit Completeness
**Category**: Compliance
**Target**: 100% of moderation actions (pass, flag, block) SHALL be recorded in the
audit trail. No moderation decision SHALL be unlogged.

## 5. User Journeys

### Journey 1: Normal gameplay — moderation is invisible

- **Trigger**: Player submits a fantasy-appropriate turn.
- **Steps**:
  1. Player types "I search the old library for clues about the missing townspeople."
  2. Input moderation classifies: `safe`, confidence 0.98, verdict `pass`.
  3. Pipeline proceeds normally. LLM generates narrative.
  4. Output moderation classifies generated content: `safe`, verdict `pass`.
  5. Narrative streams to player via SSE.
- **Outcome**: Player sees seamless narrative. Moderation added ~50ms total latency (unnoticeable).

### Journey 2: Player input is blocked

- **Trigger**: Player submits content requesting harmful real-world information.
- **Steps**:
  1. Player types content that matches `dangerous_activity` category.
  2. Input moderation classifies: `dangerous_activity`, confidence 0.92, verdict `block`.
  3. System does NOT send content to LLM.
  4. System returns a gentle redirect: "Let's take the story in a different direction.
     What else would you like to do?"
  5. Moderation event logged with content hash and category.
  6. Player resubmits with game-appropriate content. Game continues normally.
- **Outcome**: Player is gently redirected. No harmful content processed.

### Journey 3: LLM generates problematic content mid-stream

- **Trigger**: LLM begins generating content that crosses into `graphic_violence`.
- **Steps**:
  1. LLM generates tokens: "The creature tears…" (safe so far)
  2. Next tokens cross into graphic violence threshold.
  3. Output moderation detects within 2 tokens, halts token forwarding.
  4. System sends redirection: "The scene shifts as a bright light fills the room…"
  5. Sends SSE `moderation` event (metadata only, no blocked content).
  6. Full generated content saved to access-controlled audit store.
  7. Turn marked as `moderated`.
- **Outcome**: Player sees a narrative pivot. Operator can review the flagged content.

### Journey 4: Moderation service is down

- **Trigger**: Moderation model fails to load at startup.
- **Steps**:
  1. Application starts with `TTA_MODERATION_FAIL_MODE=open`.
  2. Moderation health check fails. `/api/v1/health` reports moderation as degraded.
  3. Operator alert fires.
  4. Players continue playing. All turns logged as unmoderated (WARN level).
  5. Operator restarts the moderation service. Next moderation check succeeds.
  6. System resumes normal moderation. Logs indicate moderation restored.
- **Outcome**: No gameplay interruption. Operator is aware and can review unmoderated turns.

## 6. Edge Cases & Failure Modes

| # | Scenario | Expected Behavior |
|---|----------|-------------------|
| EC-24.1 | Player input is borderline (confidence 0.5-0.7) | Verdict `flag`. Content proceeds to LLM. Operator can review. Threshold is configurable. |
| EC-24.2 | LLM generates content in a language moderation doesn't support | Flag the content for operator review. Log that language detection failed. Do not block. |
| EC-24.3 | Player uses creative spelling to evade moderation | Moderation system SHOULD apply normalization. Known evasion patterns are documented. v1 accepts some evasion risk — S19 will improve. |
| EC-24.4 | Moderation blocks content that was player-quoted (e.g., referencing a book) | False positive. Player can rephrase. Operator can tune sensitivity. NFR-24.2 limits false positive rate. |
| EC-24.5 | Very long player input (>10,000 characters) | Input is truncated to configured max length (per S10) BEFORE moderation. Moderation evaluates the truncated input. |
| EC-24.6 | Two moderation verdicts disagree (input=pass, output=block) | Output verdict takes precedence. The LLM may generate problematic content from safe input. Both verdicts are logged. |
| EC-24.7 | Player repeatedly triggers `flag` verdicts without `block` | After N flags in M minutes (FR-24.11), session is flagged for operator review. Player is not blocked but is monitored. |
| EC-24.8 | Prompt injection attempt via player input | Classified as `prompt_injection`, verdict `block`. Input is not forwarded to LLM. Event logged at high severity. |
| EC-24.9 | Moderation model disagrees with human reviewer | Operator can override via admin API (S26). Override is logged. Feedback loop improves future classifications (post-v1). |
| EC-24.10 | Content contains PII (player inadvertently types real name) | Classified as `personal_info`, verdict `flag`. Content proceeds (name might be a character name). Operator can review and escalate if real PII. |

## 7. Acceptance Criteria (Gherkin)

```gherkin
Feature: Content Moderation v1

  Scenario: AC-24.1 — Safe input passes moderation
    Given a player submits turn text "I open the door and step into the library"
    When input moderation evaluates the content
    Then the verdict is "pass"
    And the turn is processed normally

  Scenario: AC-24.2 — Harmful input is blocked
    Given a player submits turn text that matches the "self_harm" category
    When input moderation evaluates the content
    Then the verdict is "block"
    And the content is NOT sent to the LLM
    And the player receives a gentle redirect message
    And a moderation event is logged with category "self_harm"

  # [v2 — Streaming] v1 uses buffer-then-stream: output moderation runs on the
  # complete generated text before SSE delivery. Token-level stream interruption
  # requires true token-level streaming with inline moderation checks (v2).
  Scenario: AC-24.3 — Streaming output interrupted on detection
    Given a player is receiving narrative via SSE
    When the output moderation detects "graphic_violence" in the token stream
    Then token forwarding stops within 2 tokens
    And a narrative redirection message is sent to the player
    And the full generated content is saved to the audit store
    And the turn is marked as "moderated"

  Scenario: AC-24.4 — Moderation audit trail complete
    Given a turn has been processed with input and output moderation
    When the operator queries the moderation audit trail
    Then both input and output moderation events are present
    And each event contains moderation_id, turn_id, verdict, category, and confidence

  Scenario: AC-24.5 — Graceful degradation on moderation failure
    Given the moderation system is unavailable
    And TTA_MODERATION_FAIL_MODE is "open"
    When a player submits a turn
    Then the turn is processed without moderation
    And the turn is logged as "unmoderated" at WARN level
    And the /health endpoint reports moderation as degraded

  Scenario: AC-24.6 — Player does not see moderation metadata
    Given player input is blocked by moderation
    When the error response is sent to the player
    Then the response does NOT contain the moderation category
    And the response does NOT contain the confidence score
    And the response contains a friendly redirect message

  Scenario: AC-24.7 — Prompt injection blocked
    Given a player submits text containing prompt injection patterns
    When input moderation evaluates the content
    Then the verdict is "block"
    And the category is "prompt_injection"
    And the content is NOT forwarded to the LLM

  Scenario: AC-24.8 — Repeated blocks trigger session flag
    Given a player has 5 blocked actions in the last 10 minutes
    When the threshold is reached
    Then the session is flagged for operator review
    And a high-severity moderation alert is logged

  Scenario: AC-24.9 — Fantasy violence passes moderation
    Given a player submits "I swing my sword at the goblin"
    When input moderation evaluates the content
    Then the verdict is "pass"
    And the category is "safe" or "mild_violence"
    And the turn is processed normally

  Scenario: AC-24.10 — Fail-closed mode blocks unmoderated turns
    Given the moderation system is unavailable
    And TTA_MODERATION_FAIL_MODE is "closed"
    When a player submits a turn
    Then the turn is NOT processed
    And the player receives an error indicating the system is temporarily unavailable
```

### Criteria Checklist
- [ ] **AC-24.1**: Safe content passes through without delay
- [ ] **AC-24.2**: Harmful content is blocked with gentle redirect
- [ ] **AC-24.3**: Streaming output is interrupted within 2 tokens of detection
- [ ] **AC-24.4**: Complete audit trail for all moderation actions
- [ ] **AC-24.5**: System continues serving when moderation is down (fail-open)
- [ ] **AC-24.6**: Player never sees moderation internals
- [ ] **AC-24.7**: Prompt injection is detected and blocked
- [ ] **AC-24.8**: Repeated blocks trigger escalation
- [ ] **AC-24.9**: Fantasy violence is not false-positive blocked
- [ ] **AC-24.10**: Fail-closed mode blocks all turns when moderation is down

## 9. Open Questions

| # | Question | Impact | Resolution needed by |
|---|----------|--------|---------------------|
| Q-24.1 | Should v1 use an external moderation API (OpenAI Moderation, Perspective API) or an LLM-based classification prompt? | External API is simpler but adds a dependency and latency. LLM-based is more flexible but harder to tune. | Before implementation |
| Q-24.2 | What is the right confidence threshold for `block` vs. `flag`? | Too low = false positives (frustrating). Too high = missed blocks (unsafe). | Before v1 launch, needs testing with real content |
| Q-24.3 | Should moderation events include the raw content in structured logs, or only in the separate audit store? | Privacy vs. debuggability tradeoff. | Before implementation |
| Q-24.4 | How should the redirection messages be authored? Static templates? LLM-generated pivots? | Static is safer and more predictable. LLM-generated is more immersive but introduces risk. | Before implementation |

## 10. Out of Scope

- **Crisis detection and escalation to human counselors** — This is S19's domain. v1 moderation is automated only. — Deferred to S19.
- **Real-time safety scoring with confidence decay** — v1 uses simple per-turn classification. Continuous scoring is a future feature. — Deferred to S19.
- **User reporting ("flag this content")** — v1 moderation is system-initiated only. Player-initiated reporting is deferred. — Deferred.
- **Moderation model fine-tuning or feedback loops** — v1 uses off-the-shelf classification. Model improvement is post-v1. — Deferred.
- **Multi-language moderation** — v1 targets English content only. — Deferred.
- **Image or audio moderation** — TTA is text-only in v1. — N/A.
- **Moderation appeals process** — Players cannot appeal moderation decisions in v1. Operators can review. — Deferred.
- **Content age-rating** — No age-gating in v1. All content targets a general audience. — Deferred.

## Appendix

### A. Glossary

| Term | Definition |
|---|---|
| **Content moderation** | The process of evaluating content (input or output) against safety criteria before it reaches the player. |
| **Verdict** | The outcome of a moderation evaluation: pass, flag, or block. |
| **Content classification** | Categorizing content into predefined safety categories (e.g., safe, graphic_violence, self_harm). |
| **Fail-open** | A mode where the system continues operating without moderation when the moderation service is unavailable. |
| **Fail-closed** | A mode where the system blocks all content when the moderation service is unavailable. |
| **Stream interruption** | Stopping the forwarding of LLM-generated tokens to the player mid-stream when problematic content is detected. |
| **Prompt injection** | An attack where a user crafts input designed to manipulate the LLM's system prompt or override instructions. |
