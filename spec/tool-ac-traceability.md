# Introduction

This specification defines a lightweight, Python-native Acceptance Criteria (AC)
traceability standard for the TTA project. It establishes a canonical citation format,
a `@pytest.mark.spec` marker convention, a companion scanner script
(`specs/trace_acs.py`), and a `make trace` target. The goal is to make the link between
a written AC in `specs/` and the test(s) that verify it **explicit, machine-readable,
and automatically validated** — without adopting an external traceability framework.

## 1. Purpose & Scope

### Purpose

TTA agents (Claude Code, GitHub Copilot) have been organically tagging tests with AC
identifiers (e.g., `# AC-10.01` in comments, file-header coverage lists). This is
valuable but informal: there is no enforced format, no cross-reference against the
authoritative spec corpus, and no tooling that can report which of the 668 spec ACs
lack any test citation. This spec closes that gap.

### Scope

- **In scope**: citation format standard, `@pytest.mark.spec` pytest marker, scanner
  script (`specs/trace_acs.py`), `make trace` target, HTML traceability report, and
  agent instructions for future file changes.
- **Out of scope**: code coverage (line/branch coverage — that is pytest-cov's job),
  cross-referencing `src/` comments beyond informational reporting, and any changes to
  the content or structure of `specs/*.md` files.

### Audience

TTA developers, AI agents working on this codebase, and CI/CD pipelines.

### Assumptions

- Spec ACs follow the normalized format `AC-NN.NN` (two-digit spec number, dot,
  two-digit AC number, e.g., `AC-10.01`). Legacy single-digit variants (`AC-07.1`)
  must be normalized on encounter.
- `specs/index.json` is the authoritative list of all spec ACs (generated by
  `specs/index_specs.py`).
- Tests live under `tests/` and source code under `src/`.

## 2. Definitions

| Term | Definition |
|---|---|
| **AC** | Acceptance Criterion — a single testable requirement identified by a unique ID in a spec file (e.g., `AC-10.01`). |
| **AC ID** | The canonical string identifier `AC-NN.NN` where `NN` is always zero-padded to two digits. |
| **Spec** | A functional specification file in `specs/` (e.g., `specs/10-api-and-streaming.md`). |
| **Citation** | A reference to one or more AC IDs in a test function, expressed as a `@pytest.mark.spec` marker. |
| **Covered AC** | An AC whose ID appears as a `@pytest.mark.spec` argument on at least one test function. |
| **Orphan citation** | A cited AC ID that does not exist in `specs/index.json`. |
| **Uncovered AC** | An AC present in `specs/index.json` that has zero test citations. |
| **Traceability matrix** | A mapping from each AC ID to the set of test node IDs that cite it. |
| **Stub spec** | A spec with status `Stub` (S18–S22 future stubs) — excluded from uncovered-AC warnings. |
| **SDD** | Spec-Driven Development — the TTA methodology where specs are the source of truth. |
| **Spec Kit** | `github/spec-kit` — a GitHub-published SDD toolkit (see §7 for trade-off analysis). |
| **fspec** | `sengac/fspec` — a TypeScript multi-agent spec/Kanban workflow tool (see §7). |

## 3. Requirements, Constraints & Guidelines

### Citation Format

- **REQ-001**: Every test function that verifies one or more spec ACs **shall** declare
  those ACs using the `@pytest.mark.spec` decorator, e.g.:
  ```python
  @pytest.mark.spec("AC-10.01", "AC-10.03")
  def test_game_create_returns_201() -> None:
      ...
  ```
- **REQ-002**: AC IDs in markers **shall** use the normalized `AC-NN.NN` format
  (two-digit spec number, two-digit AC number). Examples: `AC-10.01`, `AC-02.07`,
  `AC-07.04`. Single-digit variants (`AC-7.4`, `AC-07.4`) are non-conforming.
- **REQ-003**: A single test function **may** cite more than one AC ID. All IDs
  **shall** be listed as separate string arguments to the marker.
- **REQ-004**: File-level header comments listing covered ACs (`# Covers AC-10.01,
  AC-10.03, ...`) **may** be retained for human readability but are **not** a
  substitute for the `@pytest.mark.spec` marker. The scanner only counts markers.

### Scanner Script

- **REQ-005**: `specs/trace_acs.py` **shall** be a standalone Python 3.12 script with
  no dependencies outside the standard library and `specs/index.json`.
- **REQ-006**: The scanner **shall** emit a traceability report in three output modes:
  - `--validate` — text summary to stdout, exits non-zero if orphan citations exist.
  - `--json` — writes `specs/trace.json` with the full traceability matrix.
  - `--html` — writes `specs/trace.html` with a sortable traceability dashboard.
- **REQ-007**: The scanner **shall** report:
  1. Total spec ACs (from `index.json`, excluding stubs).
  2. Covered ACs (have >= 1 test marker citation).
  3. Uncovered ACs (list with spec title and AC text).
  4. Orphan citations (cited in tests but not in `index.json`).
  5. Coverage percentage: `covered / (total - stubs) * 100`.
- **REQ-008**: The scanner **shall** normalize AC IDs it encounters during scanning
  (e.g., `AC-7.4` -> `AC-07.04`) before cross-referencing.

### Pytest Marker Registration

- **REQ-009**: `pyproject.toml` **shall** declare the `spec` marker in
  `[tool.pytest.ini_options].markers` to suppress `PytestUnknownMarkWarning`.
- **REQ-010**: An invalid AC ID format (non-`AC-NN.NN`) **should** emit a warning
  during collection (enforced via a `conftest.py` hook — see §4).

### Makefile

- **REQ-011**: A `make trace` target **shall** invoke
  `uv run python specs/trace_acs.py --validate` and be added to the `validate-all`
  dependency chain.
- **REQ-012**: A `make trace-html` target **shall** generate `specs/trace.html`.

### Agent Instructions

- **GUD-001**: When an AI agent creates or modifies a test function that verifies spec
  behaviour, it **should** add `@pytest.mark.spec(...)` with the applicable AC IDs.
- **GUD-002**: When an AI agent creates a new spec AC, it **should** add a corresponding
  `@pytest.mark.spec` marker to an existing or new test, or note the AC as a tracked gap.
- **GUD-003**: Agents **should not** invent AC IDs. Only IDs present in
  `specs/index.json` are valid.
- **CON-001**: The `specs/` and `tests/` directory structure **shall not** be changed
  to accommodate traceability — the scanner adapts to existing layout.
- **CON-002**: The `@pytest.mark.spec` marker **shall not** replace test execution
  markers (`unit`, `integration`, etc.) — both may coexist on the same function.

## 4. Interfaces & Data Contracts

### `@pytest.mark.spec` Marker

```python
# Single AC
@pytest.mark.spec("AC-10.01")
def test_player_create() -> None: ...

# Multiple ACs
@pytest.mark.spec("AC-10.01", "AC-10.03", "AC-10.09")
def test_full_game_flow() -> None: ...

# Coexisting with other markers
@pytest.mark.unit
@pytest.mark.spec("AC-23.04")
async def test_circuit_breaker_opens() -> None: ...
```

### `pyproject.toml` marker declaration

```toml
[tool.pytest.ini_options]
markers = [
    # ... existing markers ...
    "spec: Links test to one or more spec AC IDs (format: AC-NN.NN)",
]
```

### `conftest.py` validation hook

```python
import re
import warnings
import pytest

_AC_PATTERN = re.compile(r"^AC-\d{2}\.\d{2}$")

def pytest_collection_modifyitems(items: list[pytest.Item]) -> None:
    for item in items:
        marker = item.get_closest_marker("spec")
        if marker is None:
            continue
        for ac_id in marker.args:
            if not _AC_PATTERN.match(ac_id):
                warnings.warn(
                    f"Non-canonical AC ID '{ac_id}' in {item.nodeid}. "
                    f"Expected format: AC-NN.NN (e.g., AC-10.01)",
                    UserWarning,
                    stacklevel=2,
                )
```

### `specs/trace_acs.py` CLI contract

```
usage: trace_acs.py [--validate] [--json] [--html] [--threshold N]
                    [--tests-dir PATH] [--specs-index PATH]

Options:
  --validate          Print text summary; exit 1 if orphan citations exist
  --json              Write specs/trace.json
  --html              Write specs/trace.html
  --threshold N       Exit 1 if uncovered AC % exceeds N (default: no threshold)
  --tests-dir PATH    Root directory to scan (default: tests/)
  --specs-index PATH  Path to index.json (default: specs/index.json)
```

### `specs/trace.json` schema

```json
{
  "generated": "2026-04-21T20:00:00Z",
  "total_acs": 612,
  "stub_acs": 56,
  "covered_acs": 180,
  "uncovered_acs": 432,
  "orphan_citations": 3,
  "coverage_pct": 29.4,
  "matrix": {
    "AC-10.01": ["tests/unit/api/test_s10_ac_compliance.py::test_player_create"],
    "AC-10.02": [],
    "AC-10.03": ["tests/unit/api/test_s10_ac_compliance.py::test_error_shape"]
  },
  "orphans": ["AC-99.01", "AC-07.4"]
}
```

## 5. Acceptance Criteria

- **AC-001**: Given `@pytest.mark.spec("AC-10.01")` on a test function, when
  `trace_acs.py --json` runs, then `trace.json` includes `AC-10.01` mapped to that
  test's node ID.
- **AC-002**: Given an AC ID `AC-10.01` in `specs/index.json` and no test marker
  citing it, when `--validate` runs, then the summary lists `AC-10.01` as uncovered.
- **AC-003**: Given a marker `@pytest.mark.spec("AC-99.01")` where `AC-99.01` does not
  exist in `index.json`, when `--validate` runs, then `AC-99.01` appears in the orphan
  citations list and `trace_acs.py` exits non-zero.
- **AC-004**: Given a non-canonical ID `@pytest.mark.spec("AC-7.4")`, when pytest
  collection runs, then a `UserWarning` is emitted identifying the non-canonical format.
- **AC-005**: Given stub specs (S18-S22, status contains "Stub"), when coverage is
  calculated, then their ACs are excluded from the uncovered-AC denominator.
- **AC-006**: Given `make trace`, when invoked, then `trace_acs.py --validate` runs and
  the exit code is propagated.
- **AC-007**: Given `make trace-html`, when invoked, then `specs/trace.html` is created
  and contains a table of all ACs with their coverage status.
- **AC-008**: The scanner shall have no runtime dependencies outside Python 3.12
  stdlib and `specs/index.json`.

## 6. Test Automation Strategy

- **Test Levels**: Unit tests for `trace_acs.py` logic (marker extraction,
  normalization, orphan detection, JSON schema validation). No integration tests needed.
- **Frameworks**: `pytest` with `tmp_path` fixtures for file I/O.
- **Test Data Management**: Synthetic `index.json` fixtures with 3-5 specs; inline
  Python test files written to `tmp_path` to simulate marker presence/absence.
- **CI/CD Integration**: `make trace` added to `validate-all` — runs on every CI push.
  Initial run reports uncovered ACs as informational (no `--threshold`) until backfill
  is complete.
- **Coverage Requirements**: `trace_acs.py` itself should reach >= 90% line coverage.
- **Performance Testing**: Not applicable — script completes in < 2 s on full corpus.

## 7. Rationale & Context

### Why Not Adopt GitHub Spec Kit or fspec?

Two external SDD tools were evaluated before choosing a custom approach:

| Tool | Stars | Language | Verdict |
|---|---|---|---|
| `github/spec-kit` | 89,996 | Python CLI | Not adopted |
| `sengac/fspec` | 61 | TypeScript | Not adopted |

**GitHub Spec Kit** is a spec workflow toolkit, not a traceability tool. Its value is
in structuring the AI-assisted spec-writing process for new projects. TTA already has
an equivalent custom process (`specs/`, `index_specs.py`, SDD phases in 65 files).
Adopting Spec Kit would require migrating all 65 existing specs to its format and
replacing the `index_specs.py` toolchain — high migration cost for no functional gain.
The V-Model extension (regulatory-grade traceability matrices for IEC 62304/ISO 26262)
is far heavier than TTA's needs and targets regulated industries.

**fspec** is a TypeScript/Node.js multi-agent workflow orchestrator with Gherkin
enforcement, a Kanban TUI, and Git checkpoint management. Its traceability model links
code to Gherkin scenarios via a Kanban workflow — not compatible with TTA's Python test
suite or `pytest` infrastructure. Adding a Node.js runtime dependency to a pure Python
project solely for traceability is not justified.

**The custom approach is preferred because:**
1. TTA already has `specs/index.json` as a machine-readable AC corpus.
2. pytest's marker system is the idiomatic Python mechanism for test metadata.
3. The scanner logic is approximately 150 lines of stdlib Python — no new dependencies.
4. No format migration of existing specs or tests is required.
5. The `make trace` target integrates directly with the existing `validate-all` chain.

### Why `@pytest.mark.spec` Over Comment Scanning?

File-header comments (e.g., `# Covers AC-10.01`) are:
- Not collected by pytest — require a separate regex scan of raw source files.
- Not associated with specific test functions — a file-level comment imprecisely claims
  all functions in the file cover the same ACs.
- Not lintable or statically analyzable.

The `@pytest.mark.spec` marker is function-scoped, pytest-native, and collectible via
`item.get_closest_marker("spec")` in any conftest hook — the marker system is the
standard pytest mechanism for attaching metadata to tests.

### AC ID Normalization

The spec corpus has two historical formats due to the project timeline:
- Legacy (early S00-S09 drafts): `AC-7.4` (no zero-padding)
- Current standard: `AC-10.01`, `AC-07.04` (two-digit zero-padded)

The canonical format is `AC-NN.NN`. The scanner normalizes on read during scanning;
the conftest hook warns on write when an agent authors a non-canonical ID in a marker.

## 8. Dependencies & External Integrations

### Technology Platform Dependencies

- **PLT-001**: Python 3.12+ standard library — `re`, `json`, `pathlib`, `argparse`.
  No third-party packages required.
- **PLT-002**: pytest >= 7.0 — for `@pytest.mark` and `pytest_collection_modifyitems`
  hook. Already a declared project dependency.

### Data Dependencies

- **DAT-001**: `specs/index.json` — authoritative AC corpus. Generated by
  `specs/index_specs.py`. The scanner reads this file at runtime to build the known-AC
  set for cross-reference.

### Infrastructure Dependencies

- **INF-001**: `Makefile` — `trace` and `trace-html` targets. No new build tools.

No external systems, third-party services, or compliance dependencies are required.

## 9. Examples & Edge Cases

```python
# CONFORMING: Single AC, zero-padded
@pytest.mark.unit
@pytest.mark.spec("AC-10.01")
def test_game_create_returns_201(client: TestClient) -> None:
    resp = client.post("/api/v1/games", json={...})
    assert resp.status_code == 201


# CONFORMING: Multiple ACs on one test
@pytest.mark.spec("AC-10.09", "AC-10.10")
def test_error_response_safe(client: TestClient) -> None:
    resp = client.post("/api/v1/games", json={"bad": "input"})
    body = resp.json()
    assert "traceback" not in str(body).lower()
    assert "correlation_id" in body


# NON-CONFORMING: Single-digit format — triggers conftest UserWarning
@pytest.mark.spec("AC-7.1")        # should be "AC-07.01"
def test_something() -> None: ...


# NON-CONFORMING: Comment-only citation — invisible to trace_acs.py
# Covers AC-10.01, AC-10.03
def test_another_thing() -> None: ...


# EDGE CASE: Stub spec ACs are excluded from uncovered-AC reporting
# AC-18.01 belongs to S18 (Stub) — not counted as uncovered


# EDGE CASE: Orphan citation — spec AC removed or renumbered
@pytest.mark.spec("AC-99.01")     # not in index.json — reported as orphan
def test_deleted_feature() -> None: ...
```

## 10. Validation Criteria

1. `uv run python specs/trace_acs.py --validate` exits 0 after initial implementation
   (informational mode, no threshold set).
2. `uv run python specs/trace_acs.py --validate` exits 1 if any orphan citations exist.
3. `specs/trace.json` is valid JSON matching the schema in §4 after `--json` run.
4. `specs/trace.html` is self-contained (no external CDN URLs) after `--html` run.
5. `make validate-all` includes `make trace` and propagates its exit code.
6. Running `pytest --collect-only -q tests/` produces no `PytestUnknownMarkWarning`
   for `spec` marks after the marker is registered in `pyproject.toml`.
7. A test with `@pytest.mark.spec("AC-99.99")` causes `--validate` to exit non-zero.
8. A test with `@pytest.mark.spec("AC-7.1")` causes a `UserWarning` during collection.
